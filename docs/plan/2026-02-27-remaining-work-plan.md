# Token Router 后续任务总计划（Remaining Work Plan）

## 1. 目标

在现有后端基础上，完成 `docs/design.md` 中尚未落地的全部功能与非功能项，达到可稳定上线的 MVP 标准，并为后续前端管理台与多上游扩展提供清晰接口边界。

## 2. 当前基线（截至 2026-02-27）

已完成：
- 用户侧核心接口：注册/登录/me、修改密码、API Key 增删改查、余额、交易记录、用量查询。
- 兑换码闭环：管理员生成、用户兑换、余额入账、交易记账。
- 代理主链路：`/v1/messages`、`/v1/messages/count_tokens`、SSE 透传。
- 计费闭环：预扣、实结算、多退少补、失败退款、`usage_logs` 落库。
- 管理侧部分接口：`POST /admin/redeem-codes`、`GET /admin/redeem-codes`、`GET /admin/usage-logs`。
- 测试与构建：单测/e2e/build 通过。

未完成核心范围：
- 管理侧用户与统计接口（users/detail/update/balance/stats）。
- 多上游调度与 Bedrock/Vertex 实现。
- 非功能项：Redis 限流、统一错误格式、日志脱敏、CORS、请求体大小限制。
- 交付项：Docker Compose、Web 管理后台。

## 3. 后续任务清单（按优先级）

## P0（必须先做，保障后端闭环）

### T01 管理员用户列表与详情
- 接口：
  - `GET /admin/users`
  - `GET /admin/users/:id`
- 要求：
  - 支持分页与关键词搜索（email）。
  - 返回用户基础信息 + 当前余额 + 最近用量摘要（可先做轻量聚合）。
  - 仅管理员可访问。
- 验收：
  - e2e 覆盖分页、搜索、无权限访问。

### T02 管理员更新用户状态与权限
- 接口：
  - `PATCH /admin/users/:id`
- 要求：
  - 支持更新 `isActive`、`isAdmin`。
  - 防止管理员误操作导致“无管理员可用”（至少保留 1 个管理员，或在文档中明确约束并加保护）。
- 验收：
  - e2e 覆盖正常更新、越权、边界条件。

### T03 管理员余额调整
- 接口：
  - `PATCH /admin/users/:id/balance`
- 要求：
  - 支持增减余额。
  - 强制写入 `transactions`：`type='admin_adjust'`，包含操作说明。
  - 事务一致性：余额与流水同事务提交。
- 验收：
  - e2e 覆盖加款、扣款、余额不足拦截（若不允许扣成负数）。

### T04 平台统计
- 接口：
  - `GET /admin/stats`
- 要求：
  - 至少包含：用户总数、活跃 API Key 数、兑换码已用/未用、累计消耗（来自 `usage_logs.total_cost`）。
  - 可增加按日趋势字段，先支持近 7/30 天。
- 验收：
  - e2e 覆盖基础统计正确性。

## P1（上线稳定性）

### T05 多上游路由与故障转移
- 模块：
  - `providers`、`proxy`
- 要求：
  - Provider Registry（配置化优先级）。
  - 健康状态缓存（内存先行，后续可迁 Redis）。
  - 上游失败自动切换（有限次数，避免死循环）。
- 验收：
  - e2e 模拟主上游失败，确认自动切换并成功返回。

### T06 Bedrock / Vertex 适配器从 stub 到可用
- 要求：
  - 请求/响应字段映射到 Claude 统一结构。
  - 处理流式响应或明确暂不支持并返回一致错误。
- 验收：
  - provider 层单测 + proxy 集成测试。

### T07 统一 Anthropic 风格错误输出
- 要求：
  - 代理端点返回统一结构：
    - `{"error":{"type":"...","message":"..."}}`
  - 覆盖鉴权失败、余额不足、参数错误、上游错误。
- 验收：
  - e2e 对关键错误路径断言响应结构。

### T08 API Key 最近使用时间更新
- 要求：
  - 代理请求成功到达服务后，更新 `api_keys.last_used_at`。
- 验收：
  - e2e 请求后 `GET /api-keys` 可看到时间更新。

## P2（安全与运维能力）

### T09 Redis 限流
- 要求：
  - 按 API Key 维度限流（默认 60/min）。
  - 兑换接口独立限流（默认 5/min）。
  - 返回 429 + 标准错误结构。
- 验收：
  - e2e 压测式用例触发 429。

### T10 日志脱敏与审计
- 要求：
  - 不记录完整 API Key、密码、上游敏感 payload。
  - 关键操作（管理员改余额/改权限）记录审计日志。
- 验收：
  - 单测或集成测试验证脱敏行为（字符串匹配）。

### T11 CORS 与请求体大小限制
- 要求：
  - 仅开放管理台域名 CORS。
  - 代理端请求体限制（例如 10MB）。
- 验收：
  - e2e 覆盖超限请求返回 413（或约定状态码）。

### T12 数据库迁移与初始化策略
- 要求：
  - Prisma migration 文件化管理。
  - 补充可选 seed：初始管理员账号（读取 `ADMIN_EMAIL/ADMIN_PASSWORD`）。
- 验收：
  - 新环境可一键迁移启动。

## P3（交付形态）

### T13 Docker Compose 交付
- 要求：
  - `server + mysql + redis` 必须可一键启动。
  - 健康检查、依赖顺序、环境变量模板齐全。
- 验收：
  - 本地 `docker compose up` 后可完成注册->兑换->调用全链路。

### T14 Web 管理后台（最小可用）
- 页面范围（MVP）：
  - 登录
  - 用户管理（列表/详情/启停/调余额）
  - 兑换码管理（生成/列表）
  - 用量日志与统计
- 验收：
  - 前后端联通，可完成管理核心流程。

### T15 文档与发布清单
- 要求：
  - 更新 README、接口文档、部署文档、环境变量说明。
  - 输出上线前检查清单（配置、迁移、回滚、监控）。
- 验收：
  - 新同事按文档可在新机器完成部署与验证。

## 4. 推荐执行顺序（迭代节奏）

1. 迭代 A：T01-T04（先补齐管理端业务闭环）
2. 迭代 B：T05-T08（提升代理可靠性与兼容性）
3. 迭代 C：T09-T12（补安全与运维能力）
4. 迭代 D：T13-T15（交付与上线准备）

## 5. 每个任务统一完成标准（DoD）

- 有对应接口测试（至少 e2e 覆盖主路径 + 错误路径）。
- `pnpm -C packages/server test` 通过。
- `pnpm -C packages/server test:e2e` 通过。
- `pnpm -C packages/server build` 通过。
- 变更同步到文档（接口/配置/行为）。
